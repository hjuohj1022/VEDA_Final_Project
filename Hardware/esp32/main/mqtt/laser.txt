# Laser Control Notes

## 1. Overview

레이저 제어는 MQTT `laser/control` 토픽으로 들어온 명령을 ESP32가 받아서,
STM32로 UART 명령을 내려 실행하는 구조다.

현재 동작 흐름은 아래와 같다.

1. Server/Client -> MQTT publish to `laser/control`
2. ESP32 MQTT handler -> payload를 STM32 UART 명령으로 변환
3. ESP32 UART bridge -> STM32로 `LASER ON` 또는 `LASER OFF` 전송
4. STM32 -> 실행 결과를 UART 응답으로 반환
5. ESP32 -> STM32 응답 라인을 MQTT `motor/response`로 republish

주의:

- 레이저 전용 응답 토픽은 아직 따로 만들지 않았고, 기존 STM 응답 토픽인 `motor/response`를 그대로 사용한다.
- 따라서 서버는 `laser/control`을 publish하고, 응답은 `motor/response`를 subscribe 해서 확인해야 한다.

## 2. ESP32 side changes

이번 작업에서 반영한 내용:

- `laser/control` 토픽 추가
- `laser/control` 수신 시 `on`, `off`, `laser on`, `laser off` payload 허용
- ESP32 내부에서 STM32 UART 명령 `LASER ON`, `LASER OFF`로 변환
- STM32 응답 publish를 `QoS 1`로 상향
- UART command queue 크기 `10 -> 16`으로 증가
- `cmdUartTask` 우선순위를 frame send task보다 높게 조정
- 테스트용 Python 스크립트 `laser_test.py` 추가

관련 파일:

- `mqtt.h`
- `mqtt.c`
- `../device/cmd_uart.c`
- `../main.c`
- `laser_test.py`

## 3. MQTT interface

### Publish topic

- Topic: `laser/control`
- QoS: `1` 권장

### Allowed payloads

아래 payload는 모두 허용되도록 처리했다.

- `on`
- `off`
- `laser on`
- `laser off`
- `LASER ON`
- `LASER OFF`

권장 payload:

- `on`
- `off`

ESP32는 위 payload를 STM32 명령으로 아래처럼 변환한다.

- `on` -> `LASER ON`
- `off` -> `LASER OFF`

### Response topic

- Topic: `motor/response`
- QoS: `1` subscribe 권장

예상 가능한 응답 예:

- `LED ON`
- `LED OFF`
- `ERR`

참고:

- STM32 쪽 구현은 `LASER ON/OFF`를 받아도 응답 문자열은 `LED ON/OFF`를 반환한다.
- `ERR`가 오면 STM32가 명령을 이해하지 못했거나, 실제 펌웨어/배선/수신 타이밍에 문제가 있을 가능성이 있다.

## 4. Server side usage

서버는 아래 방식으로 사용하면 된다.

1. `laser/control`에 `on` 또는 `off`를 publish
2. `motor/response`를 subscribe 하여 결과 확인

가장 단순한 서버 규칙:

- 켜기: `laser/control` <- `on`
- 끄기: `laser/control` <- `off`
- 응답 확인: `motor/response`

### Python example

```python
import ssl
import paho.mqtt.client as mqtt
from paho.mqtt.enums import CallbackAPIVersion

BROKER = "192.168.55.200"
PORT = 8883

CA_CERT = "../certs/rootCA.crt"
CLIENT_CERT = "../certs/client-stm32.crt"
CLIENT_KEY = "../certs/client-stm32.key"

client = mqtt.Client(callback_api_version=CallbackAPIVersion.VERSION2)
client.tls_set(
    ca_certs=CA_CERT,
    certfile=CLIENT_CERT,
    keyfile=CLIENT_KEY,
    cert_reqs=ssl.CERT_REQUIRED,
    tls_version=ssl.PROTOCOL_TLSv1_2,
)
client.tls_insecure_set(True)

def on_message(client, userdata, msg):
    print(msg.topic, msg.payload.decode("utf-8", errors="replace"))

client.on_message = on_message
client.connect(BROKER, PORT, 60)
client.subscribe("motor/response", qos=1)
client.loop_start()

client.publish("laser/control", "on", qos=1)
```

### Message contract summary

- Server publish topic: `laser/control`
- Server publish payload: `on` or `off`
- ESP32 UART command to STM32: `LASER ON` or `LASER OFF`
- Server response subscribe topic: `motor/response`

## 6. Troubleshooting

### 1. 스크립트가 바로 종료됨

대부분 아래 둘 중 하나다.

- `paho-mqtt` 미설치
- 실행 인자 없이 실행했는데 입력이 전달되지 않음

확인:

```powershell
python laser_test.py on
```

### 2. `RX motor/response: ERR`

가능한 원인:

- STM32에 올라간 실제 펌웨어가 현재 소스와 다름
- STM32 UART 수신 타이밍 문제
- 배선 또는 UART 설정 문제
- 간헐적인 부하로 인해 제어/응답 경로가 밀림

확인 포인트:

- ESP32 로그에 `Topic: laser/control`이 찍히는지
- ESP32 로그에 `Queued to STM32 UART`가 찍히는지
- ESP32 로그에 `STM32 UART TX`가 찍히는지
- ESP32 로그에 `STM32 UART RX`가 찍히는지

### 3. 어떤 때는 되고 어떤 때는 안 됨

간헐성 완화를 위해 아래를 반영했다.

- STM32 응답 publish QoS를 `1`로 상향
- UART queue 크기 증가
- UART task 우선순위 상향

그래도 불안정하면 아래를 추가 확인해야 한다.

- ESP32와 STM32 간 UART 배선
- GND 공통 연결
- 실제 플래시된 ESP32/STM32 펌웨어 버전
- 열화상 frame load가 높은 상황에서의 제어 응답 지연

## 7. Recommended server behavior

서버 구현은 아래 기준으로 맞추는 것을 권장한다.

- publish topic: `laser/control`
- publish payload: `on`, `off`
- publish QoS: `1`
- response subscribe topic: `motor/response`
- response subscribe QoS: `1`
- timeout: 최소 `2 ~ 5초`

권장 처리 예:

- `on` publish
- `motor/response`에서 `LED ON` 수신 시 성공 처리
- `ERR` 수신 시 실패 처리
- timeout 시 재시도 또는 장치 상태 확인

## 8. Final note

레이저 명령은 현재 별도 장치 프로토콜을 추가하지 않고,
기존 STM32 UART command path를 재사용하는 방식으로 붙어 있다.

즉 서버에서는 새 토픽 `laser/control`만 사용하면 되고,
ESP32가 이를 STM32 UART 명령으로 브리지해 주는 구조다.
