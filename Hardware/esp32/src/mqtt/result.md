# MQTT 연결 끊김 문제 분석 및 조치 보고서

## 1. 문제 현상
- ESP32에서 38KB의 Lepton 프레임 데이터를 MQTT로 전송하는 과정에서 `Writing didn't complete in specified timeout` 및 `Poll timeout` 발생.
- 보안 연결(TLS) 핸드쉐이크 단계에서 `esp-tls: Failed to open new connection in specified timeout` 에러와 함께 연결 실패 반복.
- RSSI 수치가 -86dBm ~ -88dBm으로 매우 낮아 네트워크 패킷 유실이 심각함.

## 2. 원인 분석
1. **데이터 처리 불균형:** 
   - Teensy 4.1이 데이터를 보내는 속도가 ESP32가 MQTT로 처리하는 속도보다 빨라 버퍼 포화 발생.
2. **WiFi 신호 불량 (RSSI low):**
   - 매우 약한 신호 세기로 인해 TLS 핸드쉐이크 패킷들이 손실되어 10초 내 연결 수립 실패.
3. **리소스 및 암호화 부하:**
   - MQTTS(TLS) 사용으로 인한 CPU 연산 및 메모리 점유가 높아 통신 응답성 저하.

## 3. 조치 사항 및 결과

### [1차 조치] 전송 주기 및 딜레이 조정 (적용 완료)
- **Teensy:** 프레임 전송 후 `delay(1000)` 추가 (1fps 고정).
- **ESP32:** MQTT 청크 딜레이 조정 및 UART 태스크 최적화.
- **결과:** 초기 전송은 성공하나 지속 전송 시 타임아웃 발생.

### [2차 조치] 전송 효율화 (적용 후 롤백)
- **내용:** 청크 크기 확대(4KB), 스택 크기 조정 시도.
- **결과:** 스택 오버플로우(Stack protection fault) 발생으로 안정성을 위해 기존 1KB 청크 구조로 롤백 결정.

### [3차 조치] 보안 프로토콜 변경 (적용 완료)
- **내용:** `mqtts://` (8883) -> `mqtt://` (1883)로 변경하여 TLS 암호화 오버헤드 제거.
- **목적:** 약한 WiFi 신호 환경에서 연결 성공률을 높이고 CPU 부하를 최소화함.
- **결과:** TLS 핸드쉐이크 생략으로 즉각적인 연결 기대.

## 4. 향후 권장 사항
- **WiFi 환경 개선:** RSSI 수치를 -70dBm 이상으로 확보할 것을 강력히 권장.
- **브로커 확인:** 일반 MQTT(1883) 포트가 브로커 측에서 열려 있는지 확인 필요.
